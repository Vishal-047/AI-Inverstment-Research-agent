--------------------------------------------------------------------------------
OVERVIEW
--------------------------------------------------------------------------------
An AI-powered investment research tool that takes a company name as input,
researches the founding team using web search, scores them on a custom rubric,
and returns a structured verdict: Invest / Pass with note / Pass / Inconclusive.

The agent evaluates companies through the lens of FOUNDER & EXECUTION QUALITY,
not just market size — because early-stage outcomes are driven more by WHO is
building than WHAT is being built.

AGENT ARCHITECTURE — LangGraph StateGraph (ITERATIVE)
--------------------------------------------------------------------------------

  [START]
     ↓
  planQueriesNode      → LLM generates 3-4 founder-focused search queries.
     ↓                   On retry, reads missingInformation and generates
  executeSearchNode    → new targeted queries.
     ↓                   Tavily runs each query (basic depth, 3 results each).
  extractDataNode      → Gemini extracts structured data into Zod schema.
     ↓                   Strict prompt: "Only use provided text, do not invent."
  [CONDITIONAL EDGE]
     ├─ if dataConfidence === "low" AND retryCount < 2 → back to planQueriesNode
     └─ otherwise ↓
  scoreCompanyNode     → PURE TYPESCRIPT. No LLM. Applies weighted rubric.
     ↓                   Enforces floor rule. Computes verdict.
  decideOutcomeNode    → Gemini writes final VC memo narrative (prose only).
     ↓                   Math is already done — LLM just writes the reasoning.
  [END]

Max retries: 2 (so the agent can search up to 3 times for thin data).

--------------------------------------------------------------------------------
SCORING RUBRIC — "Founder Execution Score"
--------------------------------------------------------------------------------
Category         | Weight | What it measures
-----------------|--------|--------------------------------------------------
Track Record     |  30%   | Prior startups, exits, shutdowns & relevance
Domain-Founder   |  25%   | Does background match problem? Pivot flag?
Team Velocity    |  25%   | Key leadership hires found, hiring trend
Conviction       |  20%   | Co-founder stability, personal stake, tenure

SCORING LOGIC (scoreCompanyNode — pure TypeScript):
  Track Record:
    - Has exits                          → 9
    - Has ventures, no shutdowns         → 7
    - Has shutdowns                      → 4
    - No prior ventures found            → 3

  Domain Fit:
    - >1 relevant background credentials → 8 (base)
    - Pivot flag detected                → -3 (penalty)

  Team Velocity:
    - ≥2 key hires found                 → 8
    - 1 hire found                       → 5
    - 0 hires found                      → 3

  Conviction:
    - Co-founder stable                  → 8
    - Departures found                   → 2
    - Unclear                            → 6

WEIGHTED TOTAL = (TR×0.30) + (DF×0.25) + (VEL×0.25) + (CONV×0.20)

DECISION RULES:
  Invest          → weighted score ≥ 7.0  AND  no single category < 4
  Pass with note  → weighted score 5.5–7.0 (mixed signals)
  Pass            → weighted score < 5.5  OR  any single category < 4  ← FLOOR RULE
  Inconclusive    → dataConfidence === "low" OR missingInformation.length > 3

--------------------------------------------------------------------------------
ZOD SCHEMA — FounderSignalSchema
--------------------------------------------------------------------------------
{
  trackRecord: {
    priorVentures: [{ name, outcome: "exit|shutdown|still_running|unclear", relevance }],
    rawEvidence: string   ← direct quotes from search results
  },
  domainFit: {
    relevantBackground: string[],
    pivotFlag: boolean,
    rawEvidence: string
  },
  teamVelocity: {
    keyHiresFound: string[],
    hiringTrendNote: string,
    rawEvidence: string
  },
  conviction: {
    cofounderStability: "stable|departures_found|unclear",
    personalStakeNote: string,
    rawEvidence: string
  },
  dataConfidence: "high|medium|low",
  missingInformation: string[]  ← feeds the retry loop
}

rawEvidence in every category is critical — it grounds the UI display and
prevents the LLM from hallucinating scores without actual evidence.

--------------------------------------------------------------------------------
DESIGN DECISIONS & TRADE-OFFS
--------------------------------------------------------------------------------
1. Why Iterative (not Parallel) architecture?
   → Chosen to ensure the floor rule is well-enforced. If evidence for one
     category is thin, the agent searches again rather than incorrectly
     flagging the category as a failure.

2. Why scoring is pure TypeScript, not LLM?
   → LLMs are inconsistent with arithmetic. The rubric math MUST be
     deterministic and auditable. The LLM's job is prose, not math.

3. Why Tavily over DuckDuckGo scraping?
   → DDG scraping is fragile and easily rate-limited. Tavily returns clean
     content (not raw HTML) and is purpose-built for AI agents.

4. Why Gemini 1.5 Flash (not Pro)?
   → Flash is fast and cheap for this structured extraction task. Pro is
     overkill for extracting known schema fields from short search snippets.

5. Why rawEvidence in every category?
   → Forces grounding — the LLM cannot assign a score without citing actual
     text. Also powers the frontend: each score card shows WHY it got that score.

6. Why missingInformation[] in the schema?
   → Enables the retry loop to be targeted, not random. Instead of "search again
     generally", the agent knows specifically what to look for on the next pass.

--------------------------------------------------------------------------------
EDGE CASES TO WATCH OUT FOR
--------------------------------------------------------------------------------
  - Ambiguous company names (e.g. "Apple" → many results for wrong company)
    → Consider adding city/sector context to the initial prompt
  - Very new companies with no online presence
    → dataConfidence will be "low" → Inconclusive verdict (correct behavior)
  - Common founder names returning wrong people's LinkedIn profiles
    → rawEvidence field helps spot this in the UI
  - Tavily rate limits on free tier (1000 searches/month)
    → Add a cache layer before production deployment
  - Search returning only PR/marketing content, not real founder data
    → The missingInformation field will catch this and trigger retry
